# API Contracts

This guide defines API and DTO conventions for the NestJS backend, workers, shared contracts, and desktop-agent command payloads.

## Contract Principles

- Contracts must be explicit and versionable.
- DTOs must be JSON-compatible.
- Inputs must be validated at every trust boundary.
- Responses must not expose internal persistence entities directly.
- Breaking changes require migration strategy and compatibility notes.
- Public contract changes should update `packages/contracts`.

## Identifier Conventions

Use stable string identifiers:

```text
tenantId
userId
agentId
documentId
draftId
metadataSnapshotId
commandId
jobId
correlationId
```

Identifiers should be opaque to clients. Do not encode business meaning into IDs.

## Time Conventions

- Use ISO 8601 strings for API timestamps.
- Use UTC.
- Use explicit field names: `createdAt`, `updatedAt`, `expiresAt`, `deadlineAt`.
- Avoid ambiguous local time unless a business domain explicitly requires it.

## Error Contract

API errors should have a normalized shape:

```text
code
message
severity
source
retryable
userAction
correlationId
detailsRef
```

Do not expose stack traces, provider secrets, raw OCR text, or raw 1C diagnostics in public API responses.

## Pagination

List endpoints should define:

- page size limits;
- stable ordering;
- cursor or offset semantics;
- tenant scoping;
- filtering rules.

Large result sets must not default to unbounded queries.

## Versioning

Version when one of these changes:

- command payload shape;
- OCR artifact schema;
- metadata snapshot schema;
- draft state machine;
- write result format;
- public API route semantics.

Prefer additive changes where possible.

## Validation

DTOs must validate:

- required fields;
- enum values;
- IDs;
- date strings;
- payload size boundaries;
- array length limits;
- tenant-scoped references;
- file metadata.

Use NestJS `ValidationPipe` with whitelist and forbidden non-whitelisted properties.

## Queue Payloads

Queue payloads should contain references, not large data:

```text
tenantId
documentId
draftId
metadataSnapshotId
idempotencyKey
correlationId
```

Do not put raw documents, raw OCR text, secrets, or full metadata snapshots into queue payloads.

## Agent Command Bus Contract

`POST /api/agents/commands` creates a persisted backend-to-desktop command. It queues work for a specific tenant agent and does not execute the command inside the HTTP request.

Request rules:

- Use `payloadVersion: 1`.
- Include `tenantId`, `agentId`, `commandType`, `commandPayloadVersion`, compact JSON `payload`, `idempotencyKey`, `deadlineAt`, `correlationId`, and `createdByUserId`.
- Include `draftId` and `metadataSnapshotId` when the command targets a draft or a known 1C metadata snapshot.
- `commandId` may be supplied by an internal caller or generated by the backend.
- `maxRetries` is bounded and defaults to three.
- Do not include 1C usernames, passwords, tokens, connection strings, auth headers, or raw diagnostic payloads.

Lifecycle rules:

- New commands start as `queued`.
- Agent polling returns queued commands and marks them `delivered`.
- Terminal result submission may move a delivered or running command to `succeeded`, `failed_retryable`, or `failed_terminal`.
- Expired active commands move to `expired` before polling.

Idempotency rules:

- `(tenantId, idempotencyKey)` is unique.
- Same key plus same normalized request hash returns the existing command with `idempotencyReplay: true`.
- Same key plus a different normalized request hash returns an idempotency conflict.

## Native Browser Authentication Contract

Native desktop authentication uses an external browser flow. The desktop app starts a short-lived auth request, opens the backend-hosted login page in the system browser, and polls for completion.

Routes:

- `POST /api/auth/native/start`
- `GET /api/auth/native/login`
- `POST /api/auth/native/login`
- `POST /api/auth/native/poll`
- `GET /api/auth/me`

Start request rules:

- Use `payloadVersion: 1`.
- Include `correlationId`.
- Optionally include a compact `deviceLabel`.
- Do not include email, password, OAuth tokens, 1C credentials, or browser form fields.

Start response rules:

- Return `authRequestId`, `loginUrl`, `pollToken`, `expiresAt`, and `pollIntervalMs`.
- `loginUrl` is intended for the external system browser, not the Tauri webview.
- `pollToken` is sensitive opaque material. Do not log it, persist it outside short-lived client state, or expose it in UI copy.

Browser login rules:

- The browser form may accept email and password for the local MVP flow.
- Passwords must be hashed by the backend before persistence.
- The backend stores token hashes and auth request state hashes, not raw polling material.
- Completed native auth requests are single-use and become consumed after a successful poll.
- The successful browser page should immediately try to return/open the desktop workspace through server-side `AUTH_NATIVE_RETURN_URL` when configured.
- The successful browser page must not include the user email, access token, poll token, state, or raw auth request material in visible copy or return URLs.
- The desktop app transitions to the authenticated workspace after polling returns `authenticated`; the browser page is only a return/focus helper.

Session rules:

- `POST /api/auth/native/poll` returns `pending` until the browser login completes.
- A successful poll returns a short-lived access token and user summary.
- `GET /api/auth/me` requires a Bearer token.
- Production identity should move to Authorization Code + PKCE with deep-link or loopback callback; the MVP polling contract is a stepping stone, not the final enterprise SSO design.

## Draft Creation Contract

`POST /api/drafts` creates a persisted Automator draft for accountant review. It stores the mapped document proposal in Automator PostgreSQL only. It must not create write packages, enqueue agent commands, call workers, or write anything to 1C.

Request rules:

- Use `payloadVersion: 1`.
- Include `tenantId`, `documentId`, `metadataSnapshotId`, `schemaHash`, `documentType`, `targetResourceName`, mapped `fields`, optional `references`, `confidence`, `validationSummary`, `idempotencyKey`, `correlationId`, and `createdByUserId`.
- Do not accept caller-provided `lifecycleStatus`, `approvalStatus`, `writeStatus`, `agentCommand`, `commandId`, `writePackage`, `targetKind`, or write operation fields.
- Do not include raw documents, raw OCR text, secrets, tokens, connection strings, or 1C credentials.
- Keep mapped JSON field values compact. Large artifacts belong in object storage and should be referenced by durable IDs.
- Treat `fields` and `references` as sets for idempotency: normalize names, reject duplicates, and sort them before hashing.
- Reject malformed direct service payloads with typed `DraftCreationError` responses instead of runtime exceptions.

Response rules:

- Always return `lifecycleStatus: "needs_review"`.
- Always return `approvalStatus: "pending"`.
- Always return `writeStatus: "not_requested"`.
- Always return `requiresAccountantApproval: true`.
- Include `idempotencyReplay` to distinguish a new insert from a safe replay.

Idempotency rules:

- `(tenantId, idempotencyKey)` is unique.
- Same key plus same normalized request hash returns the existing draft.
- Same key plus a different normalized request hash returns a conflict.
- The response and errors must not echo sensitive field values. Unsafe `correlationId` and error field metadata must be dropped rather than reflected.

## Document Exception Queue Contract

`POST /api/document-exceptions` routes a problematic document into the Automator exception queue. It stores a compact, categorized exception record in Automator PostgreSQL only. It must not create drafts, enqueue workers, dispatch agent commands, create write packages, or write anything to 1C.

Request rules:

- Use `payloadVersion: 1`.
- Include `tenantId`, `documentId`, processing `stage`, one or more compact `signals`, `idempotencyKey`, `correlationId`, and `createdByUserId`.
- Optional references may include `draftId`, `metadataSnapshotId`, and `schemaHash`.
- Signals must use documented codes such as `low_document_confidence`, `nomenclature_ambiguous`, `unit_mismatch`, `metadata_gap`, or `validation_error`.
- Do not accept caller-provided `category`, `queueName`, `priority`, `status`, lifecycle fields, approval fields, write fields, command fields, or resolution fields. The backend chooses the category and queue.
- Do not include raw documents, raw OCR text, extracted full text, secrets, tokens, connection strings, or 1C credentials.

Routing rules:

- Low confidence, counterparty issues, nomenclature issues, unit mismatch, VAT mismatch, and validation errors route to `accountant_review`.
- Metadata gaps route to `admin_setup`.
- OCR failures route to `ocr_retry`.
- Unsupported document types and policy blocks route to `manual_processing`.
- The top signal is selected deterministically by severity and product risk, and the response includes the resulting `category`, `queueName`, `priority`, and `suggestedActions`.

Idempotency rules:

- `(tenantId, idempotencyKey)` is unique.
- Same key plus same normalized request hash returns the existing exception with `idempotencyReplay: true`.
- Same key plus a different normalized request hash returns a conflict.
- Audit payloads must include only identifiers, counts, selected category, queue, priority, top signal code, signal codes, and request hash. Do not store signal messages or raw values in audit payloads.

## Desktop IPC Payloads

Tauri IPC payloads should be:

- small;
- typed;
- JSON-compatible;
- free of secrets unless crossing into Rust secure storage APIs;
- validated before local side effects.

## Write Package Contracts

Write package contracts represent the boundary between an approved draft and a later 1C execution strategy. They are shared in `packages/contracts` so the backend, worker, and desktop-agent keep the same wire shape.

Rules:

- Use `payloadVersion: 1` for the first write package planner.
- Use `targetKind: "fresh-odata"` for relative OData request artifacts.
- Use `targetKind: "local-json-export"` for deterministic local JSON package artifacts.
- V1 supports only `operation: "create"`.
- The request must include the final document, active metadata object, `metadataSnapshotId`, `schemaHash`, `draftId`, `idempotencyKey`, and `correlationId`.
- The response is a plan, not an execution result.
- Artifacts must include explicit flags such as `willExecute: false`, `willWriteTo1C: false`, or `willWriteFile: false`.
- Secrets, full endpoints with credentials, auth headers, raw connection strings, and direct SQL payloads are forbidden.
- Any schema mismatch, unknown field, duplicate field, missing non-nullable field, unapproved draft, or failed validation must return a typed planner error.

## OpenAPI

- Document public API endpoints.
- Include error responses.
- Mark admin/internal endpoints clearly.
- Do not include real customer data in examples.
- Keep OpenAPI output aligned with DTOs.
